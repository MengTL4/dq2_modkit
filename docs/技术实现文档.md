# 技术实现文档

本文记录当前修改器、存档解密/加密、`data.pak` 和 `useData` 解密的实现方式。对应当前 bridge 版本：

```text
dq2_modkit/runtime/bridge/page-bridge.js  v0.2.25
```

## 设计约束

游戏是 NW.js/RPG Maker MV 系。原启动链包含完整性校验和调试限制：

- 原 `package.json` 指向 `www/index.html` 和 `www/guard.js`。
- `chromium-args` 禁用开发者工具。
- `package.json`、`www/index.html`、`www/guard.js` 不适合直接修改。
- 不能依赖远程调试端口。

因此实现路线是：

1. 保留原游戏文件不动。
2. 建立独立 NW 运行目录。
3. 由独立启动器打开原 `www/index.html`。
4. 用 NW 的 `inject_js_start` 注入 bridge。
5. GUI/CLI 通过本地文件队列和 bridge 通信。

这样游戏自己的完整性校验仍然看到原始文件，修改器逻辑只存在于额外的 `dq2_modkit` 文件中。

## 环境前置

外部工具脚本需要系统安装的 Node.js/npm。游戏目录里的 `node.dll` 只属于 NW 运行时，不能替代命令行里的 `node`、`npm`。

推荐环境：

```text
Windows 10/11
PowerShell 5+
Node.js 18+；新装优先当前 LTS
npm，随 Node.js 安装
Git，仅在需要推送仓库时使用
```

检查命令：

```powershell
node --version
npm.cmd --version
```

安装方式可以使用 Windows 包管理器：

```powershell
winget install -e --id OpenJS.NodeJS.LTS
```

或从 [Node.js 官网](https://nodejs.org/zh-cn/download) 下载安装包。安装后需要重新打开 PowerShell 让 PATH 生效。Windows PowerShell 中 `npm` 可能命中 `npm.ps1` 并受执行策略影响；工具脚本会优先调用 `npm.cmd`。

Windows 默认执行策略可能阻止 `.ps1`。长期使用建议：

```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

如果不想修改执行策略，所有脚本都可以用单次绕过方式执行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File ".\tools\launch-gui.ps1"
```

依赖安装不修改用户全局 npm 配置，而是在 `setup-runtime.ps1` 内对 `npm install` 显式传入 registry。默认 registry 是：

```text
https://registry.npmmirror.com
```

用户可通过 `-NpmRegistry` 或环境变量 `DQ2_NPM_REGISTRY` 指定其它 registry。

## 游戏根目录解析

推荐布局仍然是 `dq2_modkit` 直接放在游戏根目录下，并用项目父目录定位 `Game.exe`、`www`、`data.pak` 和存档；这种情况下用户不需要写任何配置。为了让不同用户也可以把项目放在任意位置，当前版本同时把游戏根目录抽成统一配置入口。

PowerShell 脚本通过：

```text
dq2_modkit/tools/modkit-config.ps1
```

Node/NW 脚本通过：

```text
dq2_modkit/tools/modkit-config.mjs
```

或同等的本地解析函数读取游戏目录。解析顺序是：

1. 显式参数 `-GameRoot "..."`，用于临时指定。
2. 环境变量 `DQ2_GAME_ROOT`，用于自动化或外部启动。
3. `dq2_modkit/config.local.json` 的 `gameRoot`，用于每个用户自己的固定路径。
4. 兼容旧布局：`dq2_modkit` 的父目录。

每个候选目录都必须包含 `www/index.html` 才算有效。启动脚本解析成功后会设置：

```text
DQ2_MODKIT_ROOT=<dq2_modkit 目录>
DQ2_GAME_ROOT=<游戏根目录>
```

GUI 启动 bridge 版游戏时也会把这两个环境变量传给子进程，bridge 再用它们定位 `runtime/bridge-state` 和 `www/save`。`config.local.json` 被 `.gitignore` 忽略，只提交 `config.example.json`，避免把个人 Steam 路径写进仓库。

其中 `-GameRoot` 的相对路径按当前 PowerShell 工作目录解析，方便用户从游戏目录执行 `-GameRoot "."`；`config.local.json` 和环境变量里的相对路径按项目目录解析，便于随项目移动。

## 方案选择思路

最开始的问题不是“怎么写一个按钮”，而是先判断在当前限制下哪一种修改路线最稳。这个游戏的关键特征是：

- 它是 NW.js 应用，本质上是 Chromium 页面 + Node + RPG Maker MV 运行时。
- 游戏数据和逻辑最终会进入同一个 JavaScript 页面上下文。
- 原始启动链会校验 `package.json`、`www/index.html`、`www/guard.js`。
- 远程调试和开发者工具不可用。
- 存档和资源都能离线分析，但离线编辑不能覆盖所有“当前战斗/当前进程”类需求。

因此思路是先把所有可能路线列出来，再按风险和可维护性筛选。

### 候选方案对比

| 方案 | 能做什么 | 问题 | 结论 |
| --- | --- | --- | --- |
| 直接修改 `index.html` / `package.json` / `guard.js` | 最直接注入脚本 | 触发完整性校验，且会污染原游戏文件 | 排除 |
| 开远程调试端口，用 DevTools 控制游戏页面 | 可以直接在页面上下文执行 JS | 原配置禁用调试，用户约束也明确不能走远程端口 | 排除 |
| 只做离线存档编辑器 | 能永久修改金币、背包、变量、开关、角色等已落盘数据 | 不能处理战斗倍率、技能无耗、当前地图运行态；需要关游戏、解密、编辑、加密、替换 | 保留，但只作为存档工具 |
| 直接改 `data.pak` 或 `useData` | 能改静态数据库和业务数据 | 需要重新封包，风险比查 ID 高；也可能受完整性或加载缓存影响 | 当前只做解密查看，不作为主修改路线 |
| 外部内存扫描/进程写内存 | 理论上能改运行态 | 需要指针扫描、结构识别或 native 注入；对 JS/NW 游戏反而更脆；也更容易被安全软件拦截 | 没必要，排除 |
| DLL/原生注入或图形层 overlay | 可以做外部覆盖 UI 或 hook | 对这个单机 NW/RPG Maker 场景过重；维护成本高 | 排除 |
| 浏览器扩展/content script | 思路接近页面注入 | 本地 file 页面、NW 权限、扩展加载链都不如启动器稳定；仍然要解决 Node 权限 | 不采用 |
| 独立 NW 启动器 + `inject_js_start` | 不改原文件，能在游戏页面最早期注入 JS，且能使用游戏运行时对象 | 只能影响由这个启动器打开的游戏窗口，需要维护 bridge 和 GUI 通信 | 采用 |

最终选择“运行时 bridge”的原因是：它正好贴合 NW.js 游戏的结构。游戏逻辑已经是 JavaScript 对象，不需要从进程内存里猜结构；在同一个页面上下文里调用 `$gameParty`、`$gameActors`、`DataManager`、`ConfigManager` 等对象，比写内存或反复改存档更稳。

### 为什么想到运行时修改

从解密和探针阶段已经确认几个事实：

1. 游戏使用 NW.js，页面上下文可以在合适的运行环境中拿到 Node 能力。
2. RPG Maker MV 的核心对象会在运行后挂到全局，例如 `$gameParty`、`$gameVariables`、`$gameSwitches`、`DataManager`。
3. 插件还维护了 TK 别名表，例如 `TK.$.StorageMrg`、`TK.$.DataMrg`、`TK.$.BattleMrg`。
4. 离线解密能看到数据结构，但很多功能本质上是“调用游戏已经写好的函数”，例如加物品、学技能、保存游戏。

所以自然的推导是：与其在存档里手动猜字段，不如让游戏自己执行这些动作。

例如添加物品可以直接调用：

```js
$gameParty.gainItem($dataItems[id], amount)
```

学习技能可以调用：

```js
$gameActors.actor(actorId).learnSkill(skillId)
```

保存可以调用：

```js
DataManager.saveGame(savefileId)
```

这些都是游戏原本就会使用的运行时函数，副作用、刷新、内部状态更新通常比手写存档字段更完整。

### 为什么不用远程调试也能注入

远程调试端口不可用，不代表 NW 不能加载一个独立入口。关键区别是：

- 不改原游戏入口。
- 新建自己的 NW `package.json`。
- 用自己的隐藏启动页打开原始 `www/index.html`。
- 在 `nw.Window.open()` 时使用 `inject_js_start` 指定 bridge 脚本。

这样注入发生在我们自己的启动器窗口管理逻辑里，不需要 Chrome DevTools，也不需要改原 `index.html`。

### 为什么 GUI 做外部窗口

运行时修改器有两种 UI 形态：

- 注入到游戏原生 UI。
- 独立外部 GUI。

原生 UI 的优点是沉浸、可以直接在游戏里点。但缺点很明显：

- 要理解并修改 RPG Maker 的 Scene/Window 生命周期。
- 容易和游戏菜单、输入焦点、分辨率、插件 UI 冲突。
- 每次游戏 UI 插件变化都可能影响修改器。

外部 GUI 的优点是：

- 不干扰游戏自己的画面和输入。
- 可以使用普通 HTML/CSS 做复杂列表、搜索和分页。
- 出错时只影响工具窗口，不容易把游戏 UI 搞坏。
- 通过同一个 bridge 命令队列和游戏通信，结构清楚。

因此当前采用外部 GUI，同时保留 CLI 作为调试和自动化入口。

### 各类需求对应的路线

后续遇到新功能时按这个顺序判断：

| 需求类型 | 优先路线 | 原因 |
| --- | --- | --- |
| 加物品、金币、角色、技能、变量、开关 | 运行时调用游戏函数 | 状态更新完整，立即生效 |
| 经验、金币、掉率、技能熟练度倍率 | 运行时 hook / 运行时监控 | 属于战斗结算过程，离线存档不适合 |
| 技能无耗、队伍恢复 | 运行时对象操作和资源守恒 | 需要影响当前战斗/当前队伍 |
| 查 ID、查描述、查地图、查公共事件 | 解密 `data.pak` 后读取 JSON | 静态数据最适合查表 |
| 查 useData 业务表 | 解密 `useData` | 这些数据不一定在标准 RPG Maker data 里 |
| 永久修改已落盘状态 | 离线存档编辑 | 适合关游戏后改存档 |
| 大规模改静态数据库 | 暂不作为常规路线 | 需要重新封包和额外验证 |

## 运行时修改器架构

### GUI TypeScript 构建

GUI 修改器运行在 NW.js 里，`index.html` 仍然通过 `<script src="app.js">` 加载脚本。为了让源码可维护，当前版本把业务代码迁移到 `app/gui/app.ts`，再用 TypeScript 编译出 `app/gui/app.js`：

```powershell
cd .\app\gui
npm.cmd install --registry https://registry.npmmirror.com
npm.cmd run build
```

`tools/launch-gui.ps1` 会比较 `app.ts` 和 `app.js` 的时间戳；如果 TS 源码更新过，会先安装缺失依赖并执行构建。启动 NW GUI 前，它还会检查 `output/extract/data` 中的 `_index.json`、`System.json`、`Items.json`、`Weapons.json`、`Armors.json`、`Actors.json`、`Skills.json`、`MapInfos.json`、`Troops.json`、`Enemies.json` 和 `CommonEvents.json`。这些文件缺失，或游戏目录里的 `www/data.pak` 时间戳比 `_index.json` 更新时，脚本会自动执行 `extract-data-pak.mjs`。

这样普通用户仍然只需要启动脚本，开发者也不会忘记刷新 JS 产物和 GUI 查表数据。`useData` 不是 GUI 核心列表依赖，仍由 `extract-all.ps1` 或 `extract-usedata.mjs` 手动导出。

GUI 和离线存档树形编辑器的界面都改为浅色工具型布局：白色信息面板、浅灰蓝背景、蓝色主操作、青绿色状态、琥珀色提示。这样和原先深色网格风格区分开，也更适合长时间查看列表、表单和 JSON 树。

### 启动器

`dq2_modkit/runtime/trainer/index.html` 是一个隐藏的 NW 启动页。它做三件事：

1. 计算项目目录和游戏根目录。
2. 设置 `DQ2_MODKIT_ROOT`、`DQ2_GAME_ROOT`。
3. `process.chdir(root)`，让游戏逻辑仍以原游戏根目录为工作目录。
4. `nw.Window.open()` 打开原始 `www/index.html`，并通过 `inject_js_start` 注入：

```text
dq2_modkit/runtime/bridge/page-bridge.js
```

关键点：

```js
nw.Window.open(gameUrl, {
  inject_js_start: bridgeScript
});
```

`dq2_modkit/runtime/trainer/package.json` 开启了：

```json
{
  "nodejs": true,
  "node-remote": "<all_urls>"
}
```

这样 bridge 在游戏页面里可以使用 `require("fs")`、`require("path")` 等 Node API。

### 运行时生成脚本

`app/gui`、`runtime/trainer`、`runtime/save-harness` 需要 NW 运行时文件，例如：

```text
Game.exe
nw.dll
node.dll
resources.pak
locales/
swiftshader/
```

这些文件不作为项目源码维护，而是由：

```text
dq2_modkit/tools/setup-runtime.ps1
```

从解析后的游戏根目录生成。脚本会：

1. 为三个 NW 运行目录创建 `Game.exe`、`nw.dll`、`node.dll` 等硬链接。
2. 为 `Dictionaries`、`locales`、`swiftshader` 创建目录联接。
3. 从当前 `www/js/core.jsc.pak` 和 `www/js/plugins.jsc.pak` 重新提取存档 harness 需要的 `.jsc` 字节码。
4. 检查 `tools/node_modules` 和 `runtime/save-harness/node_modules`，缺失时执行依赖安装。
5. 如果硬链接因为跨盘符等原因失败，则退回普通复制，保证项目可以放在非游戏盘。

依赖安装函数会优先调用 `npm.cmd`，避免 Windows PowerShell 命中 `npm.ps1` 后被执行策略拦截。安装命令形态是：

```powershell
npm.cmd install --omit=dev --registry <registry>
```

默认 registry 是 `npmmirror`。如果用户通过 `-NpmRegistry` 或 `DQ2_NPM_REGISTRY` 显式指定 registry，则安装失败后直接报错，方便定位企业内网源或代理问题。

这样游戏更新后不需要手工搬运行时文件，只要重新执行：

```powershell
cd "<你的 dq2_modkit 目录>"
.\tools\setup-runtime.ps1 -Force
```

启动 GUI、启动 bridge 或解密存档时，如果发现 `Game.exe` 缺失，也会自动调用 setup。`-Force` 用于游戏更新后的刷新，会删除项目目录里旧的硬链接/目录联接并重新生成，不会删除原游戏目录里的文件。字节码也会从当前版本的 `core.jsc.pak`、`plugins.jsc.pak` 重新提取，因此游戏更新后不需要手工搬运行时。

对应的清理脚本是：

```text
dq2_modkit/tools/clean-runtime.ps1
```

它只清理 `dq2_modkit` 内的生成产物：三个 NW 运行目录中的硬链接、目录联接、`runtime/save-harness` 里由当前游戏包提取的 `.jsc` 字节码、harness 探针结果，以及 `output/extract/data`、`output/extract/useData`、`output/extract/save` 解包/解密导出。脚本在删除前会检查目标路径必须位于 `dq2_modkit` 内；遇到非联接目录默认拒绝删除，只有明确列入的解包目录和依赖目录允许按普通目录递归删除。`-IncludeDependencies` 才会额外删除 `tools/node_modules` 和 `runtime/save-harness/node_modules`。

`output/backup` 和 `output/repack` 可能包含用户主动备份或准备回写的文件，清理脚本不会默认删除。

### 离线存档树形编辑器

`dq2_modkit/app/save-editor` 是额外的纯网页模块，不走 NW 运行时，也不打开游戏窗口。它和运行时修改器的边界不同：

- 不复制 `Game.exe`、`nw.dll`、`node.dll`。
- 不使用 `inject_js_start`。
- 不读写 bridge 命令队列。
- 只通过浏览器文件选择读取用户提供的 `.rpgsave` 或 `.json`，导出时走浏览器下载。

启动入口：

```text
dq2_modkit/tools/launch-save-editor.ps1
```

脚本会在 `app/save-editor` 下安装 Vite/React/jsoneditor 依赖，然后启动本地 Vite 服务。默认端口是 `5174`，可通过 `-Port` 修改；`-NoOpen` 可以只启动服务不打开浏览器；`-NpmRegistry` 和 `DQ2_NPM_REGISTRY` 控制依赖安装源。

核心库：

```text
React
jsoneditor
@msgpack/msgpack
pako
WebCrypto
```

`src/codec.ts` 在浏览器里实现两种存档格式：

- `config.rpgsave`：`base64 -> zlib inflate -> MessagePack -> JSON`。
- `global.rpgsave` / `fileN.rpgsave`：`base64 -> v2 header/ciphertext/mac -> HMAC-SHA256 -> AES-256-CBC -> zlib inflate -> MessagePack -> JSON`。

重新加密时反向执行同一套流程。v2 存档的槽位 ID 由文件名推断：`global=0`，`file1=1`，`file3=3`；文件名不标准时由界面里的槽位 ID 输入决定。

树形编辑器只暴露原始 JSON，不做金币、物品、角色等快捷业务按钮。这样它可以作为离线通用编辑器存在，和运行时修改器互不耦合。

### 复刻 Skill

`dq2_modkit/skills` 是本项目自带的 Codex skill，用来把“如何从零实现这套工具”的思路、流程和模板一起归档。它不是运行时修改器的必要依赖，删除它不会影响 GUI、bridge、解密或加密脚本运行。

组成：

```text
dq2_modkit/skills/SKILL.md
dq2_modkit/skills/agents/openai.yaml
dq2_modkit/skills/references/rebuild-playbook.md
dq2_modkit/skills/references/formats-and-contracts.md
dq2_modkit/skills/assets/dq2_modkit_template/
dq2_modkit/skills/scripts/scaffold-dq2-modkit.ps1
```

其中 `assets/dq2_modkit_template` 是干净模板，只保留源码、脚本、文档和 package 文件，不包含：

```text
Game.exe / nw.dll / node.dll 等 NW 运行时
node_modules
output
runtime/bridge-state
*.jsc 字节码
```

`scaffold-dq2-modkit.ps1` 会从 skill 所在目录反推模板路径，因此 skill 移到 `dq2_modkit/skills` 后仍可运行：

```powershell
.\dq2_modkit\skills\scripts\scaffold-dq2-modkit.ps1 -GameRoot "." -DryRun
```

如果要安装到 Codex 全局 skills 目录用于自动发现，目录名应使用 skill 名称 `dq2-modkit-builder`；当前 `dq2_modkit/skills` 位置主要用于随项目一起分发和归档。

### bridge 通信模型

GUI 和 CLI 不直接连游戏窗口，也不依赖调试端口。通信方式是本地文件队列：

- GUI/CLI 追加命令到 `commands.jsonl`。
- bridge 轮询 `commands.jsonl`，执行未处理命令。
- bridge 把结果追加到 `events.jsonl`。
- bridge 定期写 `state.json`，给 GUI 展示状态。

每条命令包含：

```json
{
  "type": "gold.add",
  "amount": 10000,
  "commandId": "时间戳-随机值",
  "ts": 1770000000000
}
```

bridge 会记录已处理 ID，避免重复执行。同时会忽略早于本次 bridge 启动时间的旧命令，避免重启后误执行历史队列。

### 状态文件

`state.json` 由 `collectState()` 生成，包含：

- bridge 版本。
- 当前页面 URL 和标题。
- 存档目录是否存在。
- 存档文件列表。
- StorageManager 是否已被路径补丁接入。
- TK 是否存在。
- 队伍对象是否可用。
- 当前金币。
- 当前地图 ID / 坐标 / 朝向。
- 当前队伍成员。
- 修改器选项。
- hook 命中统计。
- 最近错误。

GUI 读取 `state.json` 时会判断时间戳是否过期，避免把旧状态误显示为在线。

## 游戏对象解析

游戏对象可能挂在标准 RPG Maker 全局，也可能被插件放进 TK 别名表。因此 bridge 使用多路解析。

常用解析顺序：

```js
callAlias("gameParty") || window.$gameParty
callAlias("gameVariables") || window.$gameVariables
callAlias("gameSwitches") || window.$gameSwitches
callAlias("gameActors") || window.$gameActors
callAlias("gamePlayer") || window.$gamePlayer
tkValue("DataMrg") || window.DataManager
tkValue("ConfigMrg") || window.ConfigManager
```

数据表也同时查 TK 别名和标准全局：

```js
dataItems   / $dataItems
dataWeapons / $dataWeapons
dataArmors  / $dataArmors
dataSkills  / $dataSkills
dataActors  / $dataActors
dataEnemies / $dataEnemies
```

这也是之前倍率和技能无耗需要检查 TK 别名表的原因：插件可能不走标准 RPG Maker 入口。

## 存档路径补丁

bridge 会补丁 `StorageManager` 和 `TK.$.StorageMrg` 的本地路径函数：

```js
localFileDirectoryPath()
localFilePath(savefileId)
localFileExists(savefileId)
localFileBackupExists(savefileId)
isLocalMode()
```

目标目录固定到：

```text
www/save
```

这解决了通过独立 NW 启动器启动时，游戏找不到原本本地存档的问题。

## 运行时功能实现

### 金币

优先调用：

```js
$gameParty.gainGold(amount)
```

设定金币时读取当前金币，然后调用差值：

```js
gainGold(target - current)
```

如果方法不可用，则兜底写 `_gold`。

### 物品、武器、防具

GUI 从 `dq2_modkit/output/extract/data/Items.json`、`Weapons.json`、`Armors.json` 构建列表。bridge 执行时从运行时数据表取对象：

```js
const data = resolveData(kind);
const item = data[id];
$gameParty.gainItem(item, amount);
```

`kind` 允许：

```text
item
weapon
armor
```

GUI 的“全部”视图只是前端合并三类目录，真正发送时仍然拆成 `kind + id`。

### 角色

常用方法：

```js
$gameParty.addActor(id)
$gameParty.removeActor(id)
$gameActors.actor(id).recoverAll()
$gameActors.actor(id).changeLevel(level, false)
$gameActors.actor(id).gainExp(amount)
$gameActors.actor(id).learnSkill(skillId)
$gameActors.actor(id).forgetSkill(skillId)
$gameActors.actor(id).addParam(paramId, value)
```

HP / MP / TP 写入优先走：

```js
setHp()
setMp()
setTp()
```

没有方法时才写内部字段。

### 变量和开关

变量：

```js
$gameVariables.setValue(id, value)
```

开关：

```js
$gameSwitches.setValue(id, boolean)
```

GUI 的变量/开关名称来自 `System.json` 的 `variables` 和 `switches` 数组。

### 地图和公共事件

地图传送：

```js
$gamePlayer.reserveTransfer(mapId, x, y, direction, fade)
```

如果该方法不可用，则尝试：

```js
$gamePlayer.locate(x, y)
```

公共事件：

```js
$gameTemp.reserveCommonEvent(id)
```

执行后会请求地图刷新。

### 敌人图鉴

敌人 ID 来自运行时 `$dataEnemies`。执行：

```js
ConfigManager.enemyBook[id] = 1
```

同时如果 `$gameSystem` 可用，会补：

```js
$gameSystem._revealedEnemyWeaknesses[id] = true
```

最后调用：

```js
ConfigManager.save()
```

成就和称号解锁入口已经从 GUI、CLI、bridge 中移除。

### 保存

调用游戏自己的：

```js
DataManager.saveGame(savefileId)
```

这样比离线改文件更符合游戏当前运行态。

## 倍率和战斗 hook

bridge 使用 `patchMethod()` 对原型函数做运行时包装。原函数保存在 `bridge.originals`，避免重复 patch。

### 经验和金币倍率

优先作用在战斗奖励对象：

```text
BattleManager.makeRewards
BattleManager.gainRewards
BattleManager.gainExp
BattleManager.gainGold
```

bridge 会记录本场战斗原始 `_rewards`，再按倍率重算，避免同一场战斗多次触发 hook 导致重复相乘。

兜底入口：

```text
Game_Actor.prototype.gainExp
Game_Party.prototype.gainGold
```

兜底入口用于插件或 TK 别名绕过标准 `BattleManager` 结算时仍能命中。

### 掉率倍率

hook 入口：

```text
Game_Enemy.prototype.dropItemRate
Game_Enemy.prototype.makeDropItems
```

`dropItemRate` 处理标准掉率判断；`makeDropItems` 作为补足入口，用于插件绕过标准掉率函数时仍能扩大掉落。

### 脱机挂机掉落处理

脱机挂机不进入真实战斗场景，而是用地图遇敌表或固定敌群循环结算：

```text
MapXXX.json encounterList -> Troops.json -> Enemies.json
```

经验和金币优先读取运行时 `Game_Troop` 的结算函数；掉落则用数据表模拟，因为临时 `Game_Troop.makeDropItems()` 在多次脱机执行后会出现掉落上下文失效。掉落来源包括：

```text
Enemy.note <Enemy Drops> 段
Enemy.dropItems
```

装备品质来自武器、防具 note 中的稳定标签。运行时对象里有时也会带 `quality`、`_quality`、`rarity` 或 `meta.quality`，但解包数据里部分武器的 `meta.quality` 不一定可信，所以 bridge 的读取顺序是：对象显式字段 -> note 中的 `<quality:n>` -> meta 字段 -> 独立装备的基础物品。

```text
<quality:0> 劣品
<quality:1> 凡品
<quality:2> 良品
<quality:3> 精品
<quality:4> 真品
<quality:5> 极品
<quality:6> 传说
<quality:7> 传承
<quality:8> 不朽
```

`神妙`、`天工开物`、`百炼天工` 不是普通数据库品质号，而是运行时生成独立装备时可能附带的特殊词缀/打造标记。`useData` 里能看到 `神妙:*` 的效果池，例如 `神妙:轻甲掌握`、`神妙:寻宝` 等；`data.pak` 和翻译表里能看到 `天工开物`、`百炼天工` 的展示文本，但基础掉落条目不会直接带这些词缀。bridge 会在掉落物的名称、描述、note、meta 和常见词缀字段中扫描这些标签，并在结果里返回 `specialLabels`；GUI 会把它们显示在掉落名称后面。当前脱机挂机为了避免临时 `Game_Troop.makeDropItems()` 多次执行后掉落上下文失效，仍以数据表模拟掉落为主，因此只能稳定预览和结算基础掉落。如果后续要让脱机挂机自然产出带 `神妙` / `天工开物` 的随机独立装备，需要进一步复用游戏的独立装备生成函数，而不能只靠 `data.pak` 静态数据。

脱机挂机执行时先按 `blockDropQualities` 屏蔽装备掉落，再按 `autoSellQualities` 自动卖出装备。屏蔽优先级高于自动卖出；自动卖出不会写入背包，而是按 RPG Maker 常规卖价 `price / 2` 加到本次金币结算中。结果里会返回 `dropSummary`、`autoSell`、`blockedDrops` 和 `lootOptions`，GUI 用这些字段展示保留掉落、自动卖出和已屏蔽的明细。

### 技能熟练度倍率

不直接写固定熟练度。bridge 维护队伍成员技能熟练度基线，观察字段：

```text
_skillExp
_skillMasteryExp
```

如果游戏把某个技能从 `base` 增加到 `current`，bridge 按：

```text
base + (current - base) * skillRate
```

补上额外增量。这样不用知道具体插件里哪一个函数负责增加熟练度。

### 技能无耗

技能无耗最终实现为两层：

1. 函数 hook 提前拦截：

```text
Game_BattlerBase.prototype.canPaySkillCost
Game_BattlerBase.prototype.paySkillCost
Game_BattlerBase.prototype.skillMpCost
Game_BattlerBase.prototype.skillTpCost
Game_Actor.prototype.paySkillCost
Game_Actor.prototype.skillMpCost
Game_Actor.prototype.skillTpCost
```

2. 资源守恒兜底：

```text
Game_Battler.prototype.setMp
Game_Battler.prototype.setTp
Game_BattlerBase.prototype.setMp
Game_BattlerBase.prototype.setTp
Game_Actor.prototype.setMp
Game_Actor.prototype.setTp
```

开启后，bridge 记录当前队伍成员 MP/TP 基线。之后 MP/TP 只能上升，不能被技能消耗扣低。如果插件用非标准方式扣资源，定期守护也会把资源写回基线。

hook 命中会记录到：

```text
state.json -> battleStats
state.json -> rateStats
```

GUI 战斗状态行会显示最近命中的入口。

GUI 是独立 NW 应用，运行在：


它不注入游戏 UI，只通过 `dq2_modkit/runtime/bridge-state` 文件队列通信。

GUI 读取：

```text
dq2_modkit/output/extract/data/Items.json
dq2_modkit/output/extract/data/Weapons.json
dq2_modkit/output/extract/data/Armors.json

dq2_modkit/output/extract/data/CommonEvents.json
```

构建：

- 变量列表。
- 开关列表。
- 物品/武器/防具列表。
- 角色列表。
- 技能列表。
- 地图列表。
- 公共事件列表。

搜索字段包含 ID、名称、描述、note 文本等。

### 全部物品列表

GUI 内部将三类目录合并为：

```js

```

列表渲染时保留 `kind`，点击添加时发送：

```json
{ "type": "item.add", "kind": "weapon", "id": 2, "amount": 1 }
```

### 分页列表

物品、技能、角色、变量、开关、地图、公共事件列表都可能很长。GUI 不一次性渲染全部 DOM，而是按搜索结果分页：

1. 每页固定 `20` 条。
2. 搜索时先过滤完整目录，再对过滤结果分页。
3. 切换关键字时回到第一页；直接选择某个 ID 时自动跳到包含选中项的页。
4. 工具栏提供首页、上一页、下一页、末页、收起、展开和下一分类。

这样“全列表”仍可搜索，同时避免长滚动条、拖拽断开和大量 DOM 节点导致的卡顿。

脱机挂机的地图目录会读取 `MapInfos.json` 和所有 `MapXXX.json`，因此能显示全部地图。只有 `encounterList` 中存在有效 `troopId` 的地图会标记为可挂机；没有随机遇敌表的地图保留在列表中用于搜索和辨认，但 GUI 会标记“无遇敌”，执行地图挂机前也会阻止并提示改用敌群挂机。

### 图标

物品和技能图标来自游戏图标集：

```text
www/img/system/IconSet.png
```

图标文件受保护，GUI 的处理逻辑是：

1. 读取原文件。
2. 前 100 字节保持不变。
3. 100 字节之后先执行反洗牌。
4. 再逐字节异或：

```js
body[i] ^= (i % 256) ^ 90
```

得到可用 PNG 后，按 RPG Maker MV 图标规格切图：

```text
每个图标 32x32
每行 16 个
x = (iconIndex % 16) * 32
y = floor(iconIndex / 16) * 32
```

切出的图标会缓存为 data URL。

## data.pak 解密

### manifest.enc

`www/manifest.enc` 是 JSON envelope：

```json
{
  "iv": "...",
  "encryptedData": "..."
}
```

解密：

```text
AES-256-CBC
key = "f5bd74e6a64130031cd105edce551df2" 的 UTF-8 字节
iv = envelope.iv hex
ciphertext = envelope.encryptedData hex
```

解出 manifest 后得到实际资源 key。当前 key 仍为：

```text
f5bd74e6a64130031cd105edce551df2
```

注意该字符串按 32 字节 UTF-8 使用，不是 hex decode 成 16 字节。

### PAKX 外层

`www/data.pak` 结构：

```text
4 bytes   magic: PAKX
4 bytes   little-endian index envelope length
N bytes   index envelope JSON
...       sealed entries
```

索引和条目使用同一套派生 key：

```js
normalized = Buffer.from(String(key).padEnd(32, "0").slice(0, 32), "utf8")
encKey = sha256("enc:" || normalized)
macKey = sha256("mac:" || normalized)
```

认证加密格式：

```json
{
  "v": 2,
  "alg": "A256CBC-HS256",
  "iv": "...",
  "data": "...",
  "mac": "..."
}
```

索引 MAC：

```text
HMAC-SHA256(macKey, "PAKX_INDEX_V2" || iv || ciphertext)
```

条目 MAC：

```text
HMAC-SHA256(macKey, "PAKX_DATA:" + entry.path || iv || ciphertext)
```

MAC 校验通过后，用：

```text
AES-256-CBC(encKey, iv)
```

解密。

### PAK1 和旧 AES JSON

每个条目解密后得到 `PAK1` 压缩包。解压后不是最终 JSON，而是旧格式 envelope：

```json
{
  "iv": "...",
  "encryptedData": "..."
}
```

再用 manifest key 执行旧 AES：

```text
AES-256-CBC(manifestKeyUtf8, iv)
```

得到真正的数据 JSON。

导出脚本：

```powershell
cd "<你的 dq2_modkit 目录>"
node .\tools\extract-data-pak.mjs
```

输出：

```text
dq2_modkit/output/extract/data
```


解压后结构：


脚本保留：

- `.msgpack`：从 offset 20 开始的 MessagePack payload。
- `.json`：MessagePack decode 后的可读 JSON。
- `_index.json`：记录原始文件名、raw size、prefixHex、顶层 key 等。

## 存档解密

### config.rpgsave

`config.rpgsave` 格式：

```text
base64 text -> zlib inflate -> MessagePack -> JSON
```

没有 v2 AES/HMAC 外壳。

### global.rpgsave 和 file*.rpgsave

普通存档格式：

```text
base64 text
  -> magic/version/iv/length/ciphertext/mac
  -> HMAC-SHA256 校验
  -> AES-256-CBC 解密
  -> zlib inflate
  -> MessagePack
  -> JSON
```

封包结构：

```text
4 bytes   magic: 93 c1 4a 2e
1 byte    version: 02
16 bytes  iv
4 bytes   ciphertext length, little-endian
N bytes   ciphertext
32 bytes  HMAC-SHA256
```

一开始没有静态直接猜出 key，因为存档逻辑在 `TK_Expand.js.jsc`，它是 NW/V8 字节码，和游戏运行时版本绑定。普通 Node/bytenode 直接加载会出现 cached data 不匹配。

### NW harness

为了解密存档，建立了：

```text
dq2_modkit/runtime/save-harness
```

它使用游戏自带 NW 运行时，加载提取出的字节码：

```text
rpg_core.js.jsc
rpg_managers.js.jsc
rpg_objects.js.jsc
rpg_scenes.js.jsc
rpg_sprites.js.jsc
rpg_windows.js.jsc
TK_Expand.js.jsc
```

加载方式：

```js
nw.Window.get().evalNWBin(null, Buffer.from(bytecode));
```

为了让插件初始化到存档逻辑，harness 补了一些 RPG Maker/UI 全局对象占位。加载后发现存档相关对象位于：

```text
TK.$.StorageMrg
TK.$.DataMrg
```

然后重定向：

```js
TK.$.StorageMrg.localFileDirectoryPath = () => 原游戏 www/save
```

调用：

```js
TK.$.StorageMrg.load(0)
TK.$.StorageMrg.load(1)
TK.$.StorageMrg.load(3)
```

返回解密后的 MessagePack 字节，再 decode 成 JSON。

导出脚本：

```powershell
cd "<你的 dq2_modkit 目录>"
.\tools\extract-saves.ps1
```

输出：

```text
dq2_modkit/output/extract/save
```

## 存档重新加密

后续通过包装 `CryptoJS.SHA256`、`CryptoJS.HmacSHA256`、`CryptoJS.AES.decrypt`、`CryptoJS.AES.encrypt` 跟踪调用，确认了 v2 存档 key 派生和 HMAC 范围。因此现在可以用 Node 直接加密，不需要再调用游戏写入函数。

### key 派生

seed：

```js
savefileId === 0
  ? "dq2|sv2|tk_expand|RPGMV|MV|global|0"
  : `dq2|sv2|tk_expand|RPGMV|MV|save|${savefileId}`
```

派生：

```js
base = sha256(seed).hex
encKey = sha256(`${base}|enc|`)
macKey = sha256(`${base}|mac|`)
```

### file*.rpgsave / global.rpgsave

加密流程：

```text
JSON
  -> MessagePack encode
  -> zlib deflate
  -> AES-256-CBC(encKey, random iv)
  -> header + ciphertext
  -> HMAC-SHA256(macKey, header + ciphertext)
  -> base64 text
```

HMAC 范围：

```text
magic || version || iv || ciphertextLength || ciphertext
```

不包含最后的 MAC 自身。

### config.rpgsave

反向流程：

```text
config.json -> MessagePack -> zlib deflate -> base64 text
```

### 校验

`encrypt-saves.mjs` 写出文件后，会立刻按同一套逻辑解回并和输入 JSON 比对：

```text
verified=true
```

结果写入：

```text
dq2_modkit/output/repack/save/_repack-report.json
```

## 完整性校验处理

整个方案没有改原始：

```text
package.json
www/index.html
www/guard.js
```

运行时修改器使用 `dq2_modkit/runtime/trainer` 打开原页面，并注入 bridge。离线解密使用 `dq2_modkit/runtime/save-harness` 加载游戏字节码和存档函数。两者都是额外运行环境，不破坏游戏原启动链。

## 当前限制

- bridge 只影响通过 `dq2_modkit/runtime/trainer` 启动的游戏窗口，不会附加到 Steam 直接启动的旧进程。
- bridge 更新后必须重启 bridge 版游戏窗口。
- 运行时倍率和无耗是进程内 hook，关闭游戏后失效。
- 离线存档导出当前默认处理 `global`、`file1`、`file3` 和 `config`。
- 存档重新加密已经实现；`data.pak` 和 `useData` 当前只实现了解密导出，没有提供通用回写工具。
- JSON 里的 `@c`、`@a`、`@` 属于游戏序列化结构，删除或重排可能破坏存档。
